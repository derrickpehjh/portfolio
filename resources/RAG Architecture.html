03 · Design Patterns
Retrieval-Augmented Generation (RAG)

The LLM never answers from parametric memory alone. Every response is grounded in retrieved financial news chunks from Qdrant, reducing hallucination risk on time-sensitive market data.

LlamaIndex manages the VectorStoreIndex, handles chunk splitting (512 tokens, 20-token overlap), and assembles the final prompt. This separates retrieval logic from the LLM, making both independently swappable.

LlamaIndex Qdrant llama3.1:8b nomic-embed-text
Event-Driven Ingestion Pipeline

APScheduler triggers a 5-minute ingestion cycle. New article IDs are pushed to a Redis queue, decoupling ingestion from processing. The processor drains the queue independently, allowing backpressure management.

FinBERT runs in batch inference mode — articles are grouped and scored in a single forward pass, not one-by-one, reducing GPU idle time by ~70%.

Cache-Aside Pattern (Redis)

/stocks/overview — the most-called endpoint — is cached in Redis with a 60-second TTL. On miss, FastAPI queries PostgreSQL, computes the aggregated sentiment view, and writes back to cache.

This eliminates N+1 queries to PostgreSQL on every frontend poll interval (SWR refreshes every 60s), keeping the backend latency under 50ms at dashboard load.

Redis TTL 60s cache-aside SWR 60s poll
Streaming Response (Server-Sent Events)

The RAG analyst endpoint streams tokens via SSE, not a single blocking response. The frontend renders each token incrementally as it arrives, giving the user sub-second perceived latency even for long answers.

The blinking cursor and "streaming…" label provide real-time feedback. The stream terminates with a [DONE] sentinel, at which point the model ID switches from "streaming…" to "llama3.1:8b".

04 · Scaling Considerations & Possibilities
Inference Scaling
▸
vLLM as Ollama drop-in — continuous batching, PagedAttention; 5–10× throughput for concurrent RAG queries
▸
Quantized models (GGUF Q4_K_M) reduce VRAM from 16 GB to 6 GB, enabling RTX 3060 deployment
▸
Swap to Groq API for 500 tok/s vs ~30 tok/s local; zero cold-start latency
▸
Add FinBERT model cache to avoid reloading weights on each ingestion cycle
Data & Storage Scaling
▸
Qdrant distributed mode — shard collections across nodes; handles 100M+ vectors horizontally
▸
Supabase → Neon for serverless PostgreSQL with autoscaling; no idle cost
▸
Redis Cluster for high-availability cache; add Pub/Sub for live score broadcast to connected clients
▸
Hybrid search: combine vector similarity with BM25 keyword ranking in Qdrant for better precision on ticker symbols
Architecture Possibilities
▸
WebSockets replace SWR polling — push sentiment updates in real-time as FinBERT scores new articles
▸
Multi-user mode — JWT auth + per-user watchlists on Supabase; row-level security
▸
Alert engine — threshold-based sentiment alerts via email/Telegram when bull/bear score crosses ±0.5
▸
Agentic RAG — multi-step reasoning: query → reflect → re-retrieve → synthesize using LangGraph or LlamaIndex Workflows
