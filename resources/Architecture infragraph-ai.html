01 · Full System Architecture
02 · Karpathy RAG Pipeline
Key design: InfraGraph uses no vector database. Instead it follows the Karpathy wiki pattern — the LLM reads a compact wiki/index.md and selects relevant page slugs, then full .md files are loaded and passed for synthesis. The LLM acts as a semantic router rather than cosine-similarity search.
03 · API Endpoints
POST /api/search

Primary RAG endpoint. Reads wiki/index.md, calls LLM to select relevant slugs, loads full page content, synthesizes with SYNTHESIZE_QUERY_PROMPT. Returns recommendation + graph nodes/edges + evidence chain.

Also exposes GET /api/search/suggest?q= for real-time prefix-match autocomplete across titles and tags (top 5).

Karpathy RAG llama3.2 JSON mode
GET /api/graph/*

/api/graph/nodes — full graph (all pages + edges). /api/graph/nodes/{slug} — single page with related + evidence. /api/graph/discover/{slug}?depth=2 — BFS neighbor expansion for interactive graph exploration in Explorer.

Powers the React Flow canvas: clicking a node calls discover to expand its neighborhood on demand.

BFS expansion React Flow depth param
POST /api/architect/generate

Infrastructure architecture generator. Accepts constraints {cloud, scale, latency, priority}, calls ARCHITECT_PROMPT with relevant wiki pages, returns nodes + cost/complexity report with tradeoffs and risk areas.

Frontend generates Terraform HCL on the fly from the returned node list.

ARCHITECT_PROMPT cost model IaC export
GET /api/compare?tools=

Tool benchmarking endpoint. Accepts comma-separated slugs, reads wiki pages for each, extracts structured metrics — latency_ms, throughput, cost_model, ecosystem_score, failure_scenarios, tradeoffs. Frontend renders color-coded comparison table with latency bars, star ratings, and risk badges.

Allows manual tool add/remove and JSON report export.

metric extraction JSON export
POST /api/wiki/*

/ingest — raw text → INGEST_PROMPT → creates/updates wiki pages. /ingest_url — fetch URL content via fetcher.py → ingest pipeline. /ingest_bulk — fire-and-forget background task for lists of URLs. /lint — LINT_PROMPT audits for orphans, missing links, contradictions. /log — operation audit trail.

04 · Design Patterns
Karpathy Pattern (No Vector DB)

The core RAG innovation: rather than embedding chunks and running cosine similarity, InfraGraph keeps a compact wiki index (one line per page) and feeds it to the LLM, which acts as a semantic router returning relevant page slugs. Full pages are then loaded verbatim as context.

This trades retrieval precision for interpretability and zero infrastructure overhead — no embedding server, no vector store, no index rebuild on content change. The LLM's world knowledge augments the routing beyond pure keyword match.

LLM as router full-page context no embeddings no vector store
Graph-Driven Knowledge Base

Wiki pages declare explicit [[wikilinks]] in YAML frontmatter, which WikiGraph parses into a directed in-memory graph. BFS expansion lets users explore neighborhoods of any concept — clicking a node in Explorer discovers its related tools and concepts on demand.

This is fundamentally different from flat document retrieval: the graph encodes domain relationships (kafka → streaming → fault-tolerance → kafka) that guide exploration beyond the initial query.

WikiGraph BFS neighbors [[wikilinks]] React Flow
Multi-Step Decision Pipeline

InfraGraph guides users through a structured three-stage decision flow: Search (RAG query → recommendation + graph) → Compare (metric table for shortlisted tools) → Architect (constraint-driven IaC generation). Each stage is a separate page with its own API call and UI.

The Explorer automatically extracts tools from the recommendation text and pre-populates the Compare page. Architect receives constraint state from Compare via localStorage — zero manual re-entry across stages.

Search → Compare Compare → Architect localStorage state
Structured LLM Outputs (JSON Mode)

Every LLM call in InfraGraph uses Ollama JSON mode — the model is instructed to return a specific schema and the format is enforced at the API level. parser.py strips markdown code fences, extracts the first valid JSON object/array, and returns typed defaults when parsing fails.

This eliminates brittle regex parsing: the LLM output is always either valid structured data or a safe fallback — never a crash.

05 · Scaling Considerations & Possibilities
Inference Scaling
▸
vLLM as Ollama drop-in — continuous batching + PagedAttention; 5–10× throughput for concurrent queries
▸
Groq API fallback — 500 tok/s vs ~30 tok/s local; swap OllamaClient base_url to point at Groq endpoint
▸
Model upgrade path: llama3.2 → llama3.1:70b or claude-sonnet for deeper reasoning on complex constraint queries
▸
Prompt caching — wiki index is static; prepend as cached prefix to reduce tokens on every SELECT_PAGES call
Knowledge Base Scaling
▸
Vector DB upgrade path — add Qdrant/Chroma alongside Karpathy pattern for hybrid retrieval when wiki grows beyond LLM context window
▸
Git-backed wiki — version all .md edits as commits; LLM-generated pages get a traceable audit log with diff history
▸
Obsidian vault sync — wiki/pages/ is already Obsidian-compatible YAML + [[links]]; two-way sync requires no schema changes
▸
Scheduled re-ingest — APScheduler to re-fetch registered URLs and re-run INGEST_PROMPT nightly, keeping wiki pages current
Architecture Evolution
▸
Multi-agent reasoning — LangGraph pipeline: one agent retrieves, one critiques gaps, one synthesizes; increases answer depth significantly
▸
User sessions — persist Explorer state server-side; enable sharing decision URLs and team annotation of recommendations
▸
Terraform validation — pipe generated HCL through terraform validate in a subprocess and surface errors in the Architect UI
▸
Real-time cost pricing — call AWS/GCP pricing APIs at Architect generation time to replace static cost estimates with live figures
